## query（查询）

当用户询问关于 wiki 的问题时使用。不每次都从原始文档重新发现——查询基于已编译的 wiki 知识层。

### 第一步：意图路由

在搜索之前，先判断用户意图，选择最佳查询策略：

| 意图 | 检测信号 | 策略 | 上下文预算 |
|------|---------|------|-----------|
| **事实查找** | "是什么"、"定义"、"参数"、"多少" | 简单路径：单页提取 | 低（< 5K tokens） |
| **关系查询** | "关系"、"依赖"、"影响"、"连接" | 声明交叉比对 | 中（5-15K） |
| **对比查询** | "区别"、"比较"、"哪个"、"异同" | 多页收集 + 表格输出 | 中（5-15K） |
| **因果查询** | "为什么"、"导致"、"原因" | 多文档联合分析 | 高（10-25K） |
| **探索查询** | 范围模糊、开放问题 | 先给 overview，等追问 | 低起步，按需扩展 |

歧义请求默认选择匹配的最窄策略。后续追问可切换策略。

### 第二步：渐进式加载

不使用"一次加载所有候选页面"的方式。根据意图类型多轮加载：

```
R1: 读取 index.md + 最新编译报告摘要（~2K tokens）
    ↓  确定查询涉及的知识领域
R2: 读取候选页面的 frontmatter + 第一段摘要（~3K tokens）
    ↓  筛选 2-5 个最相关页面
R3: 读取候选页面的 ## 关键声明 区块（~5K tokens）
    ↓  结构化声明足够回答？→ 直接输出答案
    ↓  不够？→ 进入 R4
R4: 读取相关页面的全文（~10K tokens）
    ↓  仅在声明不足以回答时深入散文
```

**上下文预算控制**：
- 事实查找：R1→R2→R3，停在 R3（不进入全文）
- 关系/对比/因果：R1→R2→R3→R4（按需进入全文）
- 探索：R1→R2，展示 overview 后等用户指定方向

**每轮后自检**：已加载的上下文能否回答用户问题？能则停，不能则进入下一轮。

### 第三步：声明收集与交叉验证

#### 3.1 提取结构化声明

从 R3 加载的页面中提取 `## 关键声明` 区块：

```
每条声明记录：
- 声明原文
- 来源页面路径
- 来源锚点
- 声明标记（quality / claim_type / confidence，如有）
```

#### 3.2 声明质量分级

如果声明带有质量元数据，在回答中区分展示：

| 标记 | 回答中的展示 |
|------|------------|
| `quality: human_curated` | 正常展示，无标记 |
| `quality: auto_screened` | `[自动筛选]` 后缀 |
| `claim_type: direct` | 正常展示 |
| `claim_type: derived` | `[推导]` 前缀 + 推导链 |
| `confidence: low` | `⚠️ 低置信度` 前缀 |
| `status: outdated` | `⚠️ 已过时` 前缀 |

#### 3.3 编译报告交叉验证

1. 查找最新编译报告：`wiki/_compiled/report-*.md`（按日期排序，取最新）
2. 扫描 **WARN** 部分，提取与查询主题相关的矛盾
3. 扫描 **CRITICAL** 部分，检查是否有覆盖缺口
4. 如果报告不存在，静默跳过——年轻 wiki 的正常状态

#### 3.4 来源缺失处理

- wiki 无记录 → 如实说明，不编造。指出 `raw/sources/` 中对应章节供后续导入
- wiki 有记录但来源锚点缺失 → 回答但标注 `⚠️ 来源不明`
- 需要对原始来源做补充阅读 → 列出具体章节/页码范围

### 第四步：综合回答

按以下结构输出（根据意图类型调整）：

```
<核心结论，1-3 句直接回答>

<跨页面综合>（涉及 2+ 页面时包含）
各页面如何互补：
- wiki/<path>.md — <该页面对本问题的贡献>
- wiki/<path>.md — <该页面对本问题的贡献>

<矛盾呈现>（存在编译报告已知矛盾或本次新发现矛盾时包含）：
> ⚠️ 声明冲突:
> - <页面A>: <声明原文> → [来源: ...]
> - <页面B>: <声明原文> → [来源: ...]
> 建议人工确认。<如为编译报告已知矛盾，注明报告日期>

### 声明依据（涉及 2+ 页面或结构化声明时包含）
| # | 声明 | 来源页面 | 来源锚点 | 质量标记 |
|---|------|---------|---------|---------|
| 1 | <声明> | [[wiki/<path>]] | <source> | <标记> |
```

- "跨页面综合"仅在多页面时包含
- "矛盾呈现"仅在实际检出矛盾时包含——不虚构
- "声明依据"在答案来自 2+ 来源或利用了结构化声明时包含
- 对比查询输出首选表格形式

### 第五步：反馈闭环（query → compile）

查询过程可能发现编译报告未覆盖的新信息。**不静默丢弃**。

#### 5.1 新矛盾反馈

如果本次查询发现了编译报告未记录的矛盾声明：
1. 在回答中显式呈现
2. 如果 `wiki/_compiled/` 目录存在且最新报告为今天的日期，追加矛盾到报告
3. 如果最新报告不是今天的，创建新报告或追加到"## Query 发现"区块：

```markdown
## Query 发现 — [2026-04-29]

> ⚠️ 新发现矛盾（来自 query，非 compile 扫描）:
> - [[wiki/a.md]]: "时钟源为 PCLKB" → [来源: manual p.134]
> - [[wiki/b.md]]: "时钟源为外部晶振" → [来源: datasheet p.22]
> 建议运行 compile 重新扫描确认。
```

#### 5.2 过时声明反馈

如果查询发现某页面声明与更新的来源不一致：
1. 在回答中标注
2. 建议运行 `compile` 或手动刷新该页面

### 第六步：保存与刷新

1. 如果答案值得保存，提供选项：
   - `wiki/analyses/<slug>.md` — 分析结果
   - `wiki/comparisons/<slug>.md` — 对比结果
2. 如果答案揭示了过时或错误内容，提供刷新受影响页面的选项
3. 如果用户选择保存，写入后更新 `log.md`

## 多轮追问协议

### 识别追问

满足以下任一条件视为追问：
- 使用指代词引用之前的回答（"这个"、"那个"、"它"、"这些"）
- 在之前回答的基础上要求细化或展开（"具体说说X"、"展开讲讲Y"）
- 质疑或交叉验证之前的回答（"但Z页面不是这样说的"、"这两个矛盾吧"）
- 与上一轮查询共享相同的核心主题词

### 追问处理

1. **不重新执行完整搜索。** 重用本轮已读取的页面
2. 仅针对追问引入的新角度进行补充搜索（0-2 页）
3. 如果追问提出新角度但当前轮次信息不足，从 R2 重新开始
4. 在回答中明确引用前一轮上下文（例如"根据刚才查看的 adc.md..."）
5. 如果追问揭示了编译报告未覆盖的新矛盾，执行第五步反馈闭环
6. 连续追问同一主题超过 3 轮后，建议保存为 `wiki/analyses/<slug>.md`

### 会话上下文

在单次对话中隐式维护（不写入文件）：
- 本轮已读取的页面列表
- 本轮已引用的关键声明
- 本轮已检出的矛盾
- 当前加载轮次（R1/R2/R3/R4）

## 简单查询（快速路径）

事实查找意图可使用简化流程：
1. 读取 `index.md`
2. 定位最佳匹配页面（1 页）
3. 读取该页面的 `## 关键声明` 区块或首段
4. 直接回答，引用来源：
```
<直接回答>
→ 来源: wiki/<path>.md
```
5. 跳到第六步（保存与刷新），跳过第四步的完整结构

## 输出结构

无文件写入（除非用户同意保存）。
如果用户同意保存：
```
wiki/analyses/<slug>.md     ← 保存的分析结果
wiki/comparisons/<slug>.md   ← 保存的对比结果
```
如果新矛盾写回编译报告：
```
wiki/_compiled/report-YYYY-MM-DD.md  ← 追加 "## Query 发现" 区块
```
